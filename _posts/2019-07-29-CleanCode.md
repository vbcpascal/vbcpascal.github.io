# 《代码整洁之道》阅读笔记

*Clean Code: A Handbook of Agile Software Craftsmanship* 是 Robert C. Martin 的书籍。本书提出“代码质量与其整洁度成正比”，以Java语言为例，并从命名到重构多个角度讲述代码整洁之道。以下为笔者提炼的一些内容。

![](/images/reading/cleancode.png)



## 第一章	整洁代码

> 沃德原则（原书36页）：如果每个例程都让你感到深合己意，那就是整洁代码。



## 第二章	有意义的命名

总结来讲，作者不断强调**明确**是王道。变量命名要准确、明确，而不惧怕变量名较长。

**具体内容**

1. 名副其实：命名要**准确**，不能模糊。如`theList`无法表示任何东西；

2. 避免误导：

   避免使用与本意相悖的词。如不要用`accountList`，除非它真的是`List`类型；

   提防使用不同处较小的名称。如`XYZControllerForEfficientHandlingOfStrings`和`XYZControllerForEfficientStorageOfStrings`（也便于后文说的检索）；

   不要使用`l`和`O`做变量名；

3. 做有意义的区分（不要冗余）：

   不要用`a1`、`a2`、`a3`，完全没有提供信息；

   不要做没有意义的区分。如`ProductInfo`、`ProductData`、`Product`意义并无区别；

   不要有废话。如`Variable`不该出现在变量名中；

4. 读的出来；

5. 使用可搜索的名称，便于检索：

   单字母名称仅用于短方法中的本地变量；

6. 避免使用编码：无需前缀后缀（看起来像历史遗留问题？）。如`m_dsc`；

7. 避免思维映射：不应当让读者在脑中把你的名称翻译成他们熟悉的名称。如`i`、`j`；

8. 类名应该是名词、名词短语；

   方法名应该是动词、动词短语；

9. 别扮可爱：就是别开玩笑；

10. 每个概念对应一个词。如`controller`、`manager`、`driver`不要混用；

11. 不要用双关语。



## 第三章	函数

1. 短小：不该长于一屏；

2. 只做一件事：看看能否拆分出一个函数；

3. 每个函数一个抽象层级：

   向下规则：让每个函数后面都跟着位于下一抽象层级的函数；

4. 函数参数尽可能少；

5. 函数应该无副作用：只承诺做一件事；

6. 使用异常替代返回错误码；

7. 不要重复。

> Edsger Dijkstra 的结构化编程原则：每个函数、函数中的代码块都应该有一个入口、一个出口，意味着每个函数中只有一条`return`语句。循环中不能有`break`、`continue`，而且永永远远不能有`goto`。



## 第四章	注释

> 别给糟糕的代码写注释，重新写吧。		—— Brian W.Kernighan 与 P.J. Plaugher

1. 尽可能用好的代码阐述，而非注释；

2. （作者认为的）好的注释如下：

   - 法律信息；

   - 提供信息的注释。如解释了某个抽象方法的返回值；

   - 对意图的解释；

   - 警示；

   - TODO注释：记得删除；

   - 放大看起来不合理事物的重要性；

3. 坏的注释举了很多例子，大多都是指“废话”：

   - 署名、日志、位置标记（如一串`/////`）等；
   - 删除注释的代码！

作者认为有时文档的注释就是典型的“废话”，如下，可是又能怎么办呢。

``` java
/** The name */
private String name;
```



## 第五章	格式

1. 垂直格式

   每个文件尽量200行，最多500行；

   用空白行分隔代码组；

   关系密切的概念相互靠近；

2. 横向格式
   
   操作符前后空格；
   
   缩进；
   
3. 遵循团队规则；
