## 绪言

*Clean Code: A Handbook of Agile Software Craftsmanship* 是 Robert C. Martin 的书籍。本书提出“代码质量与其整洁度成正比”，以Java语言为例，并从命名到重构多个角度讲述代码整洁之道。以下为笔者提炼的一些内容。

![](/images/reading/cleancode.png)

按照作者的叙述，本书可分为三部分，第一部分（1~13章）介绍编写整洁代码的原则、模式和实践；第二部分（14章~16章）包括若干个复杂性不断增加的案例研究；第三部分（17章）是从案例中得到的启示和灵感。

笔者将第一部分再次划分为两个部分（其实是因为一个文档太长了QAQ），分割后结果如下：

第一部分（1~10章）着重于一些基础问题，如变量、函数、类，适合包括 C++ 程序员在内的朋友阅读，一些与 Java 不一致的内容可以忽略；

第二部分（11章~13章）着重于相对大型的工程，同时偏向针对 Java 进行说明。考虑到笔者没有学习过 Java，相关内容可能理解不够深刻，会在后续时间补充；

第三部分（14章~16章）着重于代码案例分析，由于书上大部分内容都是代码，这一部分随缘更新~

第四部分（17章）单独一章为一部分，是对所有上述内容的总结所形成的清单，实在不想看的可以直接来看这一章。



## 第一章 整洁代码

混乱的代码会大幅度降低你的工作效率。

> 沃德原则（原书36页）：如果每个例程都让你感到深合己意，那就是整洁代码。



## 第二章 有意义的命名

本章主要讨论了变量的命名，作者不断强调**明确**是王道。变量命名要准确、明确，而不惧怕变量名较长。

**具体内容**

1. 名副其实：命名要**准确**，不能模糊。如`theList`无法表示任何东西；

2. 避免误导：

   避免使用与本意相悖的词。如不要用`accountList`，除非它真的是`List`类型；

   提防使用不同处较小的名称。如`XYZControllerForEfficientHandlingOfStrings`和`XYZControllerForEfficientStorageOfStrings`（也便于后文说的检索）；

   不要使用`l`和`O`做变量名；

3. 做有意义的区分（不要冗余）：

   不要用`a1`、`a2`、`a3`，完全没有提供信息；

   不要做没有意义的区分。如`ProductInfo`、`ProductData`、`Product`意义并无区别；

   不要有废话。如`Variable`不该出现在变量名中；

4. 读的出来；

5. 使用可搜索的名称，便于检索：

   单字母名称仅用于短方法中的本地变量；

6. 避免使用编码：无需前缀后缀（看起来像历史遗留问题？）。如`m_dsc`；

7. 避免思维映射：不应当让读者在脑中把你的名称翻译成他们熟悉的名称。如`i`、`j`；

8. 类名应该是名词、名词短语；

   方法名应该是动词、动词短语；

9. 别扮可爱：就是别开玩笑；

10. 每个概念对应一个词。如`controller`、`manager`、`driver`不要混用；

11. 不要用双关语。



## 第三章 函数

本章讨论函数的写法，包括职能、参数，作者强调“一个函数就应该只做一件事”。如果本段中代码过于冗长，就应该按照职责进行拆分。

**具体内容**

1. 短小：不该长于一屏；

2. 只做一件事：看看能否拆分出一个函数；

3. 每个函数一个抽象层级：

   向下规则：让每个函数后面都跟着位于下一抽象层级的函数；

4. 函数参数尽可能少；

5. 函数应该无副作用：只承诺做一件事；

6. 使用异常替代返回错误码；

7. 不要重复。

> Edsger Dijkstra 的结构化编程原则：每个函数、函数中的代码块都应该有一个入口、一个出口，意味着每个函数中只有一条`return`语句。循环中不能有`break`、`continue`，而且永永远远不能有`goto`。



## 第四章 注释

> 别给糟糕的代码写注释，重新写吧。		—— Brian W.Kernighan 与 P.J. Plaugher

本章讨论注释的写法，作者认为注释的恰当用法是弥补我们在用代码表达意图时遭遇的**失败**，并给出了一些作者观点下好的、不好的注释。

1. 尽可能用好的代码阐述，而非注释；

2. （作者认为的）好的注释如下：

   - 法律信息；

   - 提供信息的注释。如解释了某个抽象方法的返回值；

   - 对意图的解释；

   - 警示；

   - TODO注释：记得删除；

   - 放大看起来不合理事物的重要性；

3. 坏的注释举了很多例子，大多都是指“废话”：

   - 署名、日志、位置标记（如一串`/////`）等；
   - 删除注释的代码！

作者认为一些文档的注释就是典型的“废话”。

``` java
/** The name */
private String name;
```



## 第五章 格式

本文主要讨论了代码格式化的问题，当然现在很多问题交给IDE去做了。

1. 垂直格式

   每个文件尽量200行，最多500行；

   用空白行分隔代码组；

   关系密切的概念相互靠近；

2. 横向格式
   
   操作符前后空格；
   
   缩进；
   
3. 遵循团队规则；



## 第六章 对象和数据结构

本章的主要观点就是下面的第二点，但同时作者表达“优秀的软件开发者不带成见地了解这种情形，并依据手边工作的性质选择其中一种手段”。

1. 数据抽象：不暴露数据细节，以抽象形态表述数据；

2. 数据、对象的反对称性：

   *过程式代码（数据结构）便于在不改动既有数据结构下添加新函数；*

   *面向对象便于在不改动既有函数下添加新的类；*

3. 得墨忒定律：类`C`的方法`f`只应该调用以下对象的方法：

   - `C`；
   - 由`f`创建的对象；
   - 作为参数传递给`f`的对象；
   - 由`C`的实体变量持有的对象；

4. 数据传送对象：**DTO**（Data Transfer Objects）是只有公共变量，没有函数了类。



## 第七章 错误处理

> 错误处理很重要，但是如果他搞乱了代码逻辑，就是错误的做法。

1. 使用异常而非返回码；
2. 先写 Try-Catch-Finally 语句；
3. 使用不可控异常；
4. 给出异常发生的环境声明；
5. 依调用者需要定义异常类；
6. 任何时候**不要返回、传递null值**。



## 第八章 边界

本章介绍整合外来代码，例如购买第三方程序包或使用开放源代码。

1. 浏览和学习边界：

   **学习性测试：**不要在生产代码中试验新东西，而是编写测试来遍览和理解第三方代码。优点为毫无成本、有正面回报、确保合己意；

2. 学习log4j：[**Log4j**](https://zh.wikipedia.org/wiki/Log4j)是一个基于Java的日志记录工具，具体使用方法不在此赘述；

3. 使用尚不存在的代码：（为了工作不受阻碍）先定义自己要用的接口，再在API设计后，通过编写Adapter跨接；

4. 整齐的边界：通过接口管理第三方边界，使用适配器模式将第三方提供的接口转换为我们的接口。



## 第九章 单元测试

> 在计算机编程中，**单元测试**（英语：Unit Testing）是针对程序模块（软件设计的最小单位）来进行正确性检验的**测试**工作。

1. [TDD](<https://en.wikipedia.org/wiki/Test-driven_development>)三定律：

   - 在编写不能通过的单元测试前，不可编写生产代码；
   - 只可编写刚好无法通过的单元测试，不能编译也算不通过；
   - 只可编写刚好足以通过当前失败测试的生产代码；

2. 保持测试整洁：同时保证生产代码和测试代码的整洁，**测试代码和生产代码一样重要**；

3. 整洁的测试：可读性（明确）、可读性（简洁）、可读性（足够的表达力）。

   测试要呈现**构造-操作-检验**模式，分别用于构造测试数据、操作测试数据、检验操作是否达到期望；

   双重标准：例如嵌入式的系统，可能不必在测试环境中限制内存资源等；

4. 每个测试一个断言；

5. **F.I.R.S.T原则：**

   Fast（快速）测试应该足够快；

   Independent（独立）测试应该相互独立；

   Repeatable（可重复）应该可以在任何环境中通过；

   Self-Validating（自足验证）应该有布尔值输出，而非通过日志；

   Timely（及时）测试应该及时编写。
   
   

## 第十章 类

本章在函数之外的更高层面讨论类的组织。与函数类似，类应该短小并且权责单一。

1. 类的组织：

   按照Java约定，类的组织方式如下

   - 公共静态常量 -> 私有静态变量 -> 私有实体变量（很少有公共变量）

   - 变量列表 -> (公共函数 -> 该函数调用的私有工具函数)

2. 类应该短小：

   **单一权责原则**（SRP）：类或模块有且只有一条加以修改的理由；

   **内聚**：方法、变量相互依赖性强，在那个方法使用每个实体变量时，内聚性最大；

   *因此，系统应该由许多短小的类而不是少量巨大的类组成。*

3. 为了修改而组织：降低修改的风险。

   **开放-闭合原则**（OCP）：类应该对扩展开放，对修改封闭。（通过子类化的手段）

   **依赖倒置原则**（DIP）：类应该依赖于抽象而不是依赖于具体细节。
