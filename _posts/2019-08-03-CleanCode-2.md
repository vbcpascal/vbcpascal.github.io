## 第十一章 系统

> 复杂要人命。它消磨开发者的生命，让产品难以规划、构建和测试。	—— Ray Ozzie

本章主要讨论在较高的抽象层级上——系统层级——保持整洁。系统应该是整洁的，侵害性架构会湮灭领域逻辑，冲击敏捷能力。

1. 将系统的构造与使用分开；

   *软件系统应将起始过程和起始过程之后的运行时逻辑分离开，在起始过程中构建应用对象，也会存在互相缠结的依赖关系。*

   - 分解`main`：将全部构造过程搬迁至`main`中；
   - 抽象工厂模式：自行控制创建，但是创建细节与程序分开；
   - **依赖注入**（DI）：是控制反转（IoC）在依赖管理中的一种应用手段；

2. 扩容：只实现今天的用户故事，然后重构。“一开始就做对系统”纯属神话。

   *与物理系统相比软件系统比较独特。他们的架构都可以递增式地增长，只要我们持续将关注面恰当地切分。*

   - 横贯式关注面：**AOP**（面向切面编程）中，被称为aspect的模块构造指明了系统中那些点的行为会以某种一致的方式修改，从而支持某种特定的场景。
   
3. AOP：
   
   - Java 代理：Java代理可以适用于简单的AOP情况。
   - 纯 Java AOP 框架：在数个Java框架中，代理都是内嵌的，如 Spring AOP 和 JBoss AOP 等，从而能够以纯 Java 代码实现AOP。
   - AspectJ：通过方面来实现关注面切分的功能最全的工具是 AspectJ 语言，一种提供“一流的”将方面作为模块构造处理支持的 Java 扩展。
   
4. 测试驱动系统架构：

   *最佳的系统架构由模块化的关注面领域组成，每个关注面均用纯 Java（或其他语言）对象实现。不同的领域之间用最不具有侵害性的方面或类工具整合起来。这种架构能测试驱动，就像代码一样。*

5. 优化决策：

   *拥有模块化关注面的 POJO（Plain-Old Java Object，旧时的 Java 对象）系统提供的敏捷能力，允许我们基于最新的知识做出优化的、时机刚好的决策。决策的复杂性也降低了。*

6. 明智使用添加了可论性价值的标准：

   *有了标准，就更复用想法和组件、雇用有相关经验的人才、封装好点子，以及将组件连接起来。*

7. 系统需要领域特定语言：

   *DSL（Domain-Specific Language，领域特定语言）允许所有抽象层级和应用程序中的所有领域，从高级策略到底层细节，使用 POJO 来表达。*

>**AOP**：
>
>**面向切面的程序设计**（Aspect-oriented programming，AOP，又译作**面向方面的程序设计**、**剖面导向程序设计**）是计算机科学中的一种编程范型，旨在将**横切关注点**与业务主体进行进一步分离，以提高程序代码的模块化程度。通过在现有代码基础上增加额外的**通知**（Advice）机制，能够对被声明为“**切点**（Pointcut）”的代码块进行统一管理与装饰，如“对所有方法名以`set*`开头的方法添加后台日志”。该思想使得开发人员能够将与代码核心业务逻辑关系不那么密切的功能（如日志功能）添加至程序中，同时又不降低业务代码的可读性。面向切面的程序设计思想也是面向切面软件开发的基础。



## 第十二章 迭进

Kent Beck 关于**简单设计**的四条原则，对于创建具有良好设计的软件有着莫大的帮助。（以下按重要程度排序）

1. 运行所有测试：遵循有关编写测试并持续运行测试的简单、明确的规则，系统就会更贴近OO低耦合度、高内聚度的目标。编写测试引致更好的设计。
2. 不可重复：重复是良好设计系统的大敌。**模板方法模式**是一种移除高层重复的通用技巧。
3. 表达力：软件项目的主要成本在于长期维护，因此代码应该清晰的表达作者的意图。
4. 尽可能少的类和方法。

> **模板方法模型**是一种行为设计模型。**模板方法**是一个定义在父类别的方法，在**模板方法**中会呼叫多个定义在父类别的其他方法，而这些方法有可能只是抽象方法并没有实作，**模板方法**仅决定这些抽象方法的执行顺序，这些抽象方法的实作由子类别负责，并且子类别不允许覆写模板方法。



## 第十三章 并发

本章讨论了并发程序的设计，但正如作者所说，编写整洁的并发程序很难，值得用一本书做讨论。本章只给出了一个概览。

**并发的目的：**

解耦**目的**和**时机**，可以明显改进应用程序的吞吐量和结构。

- 并发会在性能和编写额外代码上增加一些开销；
- 正确的并发是复杂的，即使对于简单问题也是如此；
- 并发缺陷并非总能重现，所以常常被看作偶发事件而忽略；
- 并发常常需要对设计策略的根本性修改。

**并发的挑战：**

线程执行的有些路径会产生错误的结果；

**并发防御原则：**

1. 单一权责原则（SRP）：

   - 并发相关代码有自己的开发、修改、调优的生命周期；
   
   - 并发相关代码有自己要对付的挑战，和非并发相关代码不同；
   
   - 即使没有周边应用程序增加负担，写得不好的并发代码可能出错的方式数量也已经足具挑战性。
   
   **建议：**分离并发相关代码与其他代码；
   
2. 推论：限制数据作用域：

   **建议：**谨记数据封装；严格限制对可能被共享的数据的访问。

3. 推论：使用数据复本：

   **建议：**某些情形下，避免共享数据，复制对象，以只读方式对待。

4. 推论：线程尽可能地独立：

   **建议：**尝试将数据分解到可被独立的进程（可能在不同处理器上）操作的独立子集。

**了解 Java 库：**

- 使用类库提供的线程安全群集；
- 使用 executor 框架执行无关任务；
- 尽可能使用非锁定解决方案；
- 有几个类并不是线程安全的。

**了解执行模型：**

- 生产者-消费者模型；
- 读者-作者模型；
- 宴席哲学家；

- **建议：**学习基础算法，理解其解决方案。

**警惕同步方法之间的依赖：**

- **建议：**避免使用一个共享对象的多个方法。如果不得不时，有以下三种手段；
- 基于客户端的稳定——客户端代码在调用第一个方法前锁定服务端，确保锁的范围覆盖了调用最后一个方法的代码；
- 基于服务端的稳定——在服务端内创建锁定服务端的方法，调用所有方法，然后解锁。让客户端代码调用新方法；
- 适配服务端——创建执行锁定的中间层。

**保持同步区域微笑：**

- **建议：**尽可能减小同步区域。

**很难编写正确的关闭代码：**

- **建议：**尽早考虑关闭问题，尽早令其工作正常。这会花费比你预期更多的时间。检视既有算法，因为这可能会比想象中难得多。

**测试线程代码：**

- **建议：**编写有潜力暴露问题的测试。以下是一些建议：

1. 将伪失败看作可能的线路问题：

   **建议：**不要将系统错误归咎于偶发事件。

2. 先使非线程代码可工作：

   **建议：**不要同时追踪非线程缺陷和线程缺陷。确保代码在线程之外可排序。

3. 编写可插拔的线程代码：

   **建议：**编写可在数个配置下运行的代码。

4. 编写可调整的线程代码。（线程数量可调整）

5. 运行多于处理器数量的线程。（任务交换越频繁，越有可能找到错过临界区或导致死锁的代码）

6. 在不同平台上运行：

   **建议：**尽早并经常地在所有目标平台上运行线程代码。

7. 装置试错代码。

8. 硬编码。

9. 自动化。



## 第十四章 逐步改进

从本章开始为本书的第三部分。这一部分一阅读代码为主，考虑到这些代码又臭又长，便只整理所得结论与要点，不会罗列书上的代码。

本章给出了 Args 实现的例子，说明一个开始还不错，规模扩大后即出问题的模块，以及对这样的模块是如何进行重构而变得整洁的。

- 混乱是逐渐产生的。例如添加对类型的支持，使得小混乱变成大混乱。
  - 在出现以上状况时，及时进行调整、重构。
- 毁坏程序最好的方法之一就是以改进之名大动其结构，会导致很多程序难以恢复回来。
  - 为避免以上情况，采用测试驱动开发的规程。
